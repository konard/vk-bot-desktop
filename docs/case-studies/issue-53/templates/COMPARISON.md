## Shared bugs

| Bug                                                                                                                                                                                                                                                      | Repos affected                                                             | File:line                                                                                                                                                               | Proposed fix                                                                                                                                                                                                                                   |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Silent push failure.** `command-stream`'s `await $\`...\``does NOT throw on non-zero exit by default, so a rejected`git push` (non-fast-forward) silently reports success, the version-bump commit is lost, and downstream jobs build off a stale SHA. | vk-bot-desktop, **js** template                                            | `scripts/version-and-commit.mjs:263-271` (both repos, identical)                                                                                                        | Either (a) enable `command-stream` shell-error mode, (b) capture exit status (`const r = await $\`git push…\`.run({ capture: true }); if (r.code !== 0) throw…`), or (c) wrap in a retry loop with `git pull --rebase` like the Rust template. |
| **Rebase without uncommitted-changes pre-check.** `git rebase origin/main` aborts with "cannot rebase: You have unstaged changes" if any tracked file is dirty, but the script does not run `git status --porcelain` first.                              | vk-bot-desktop, **js** template, **python** template                       | js/vk: `scripts/version-and-commit.mjs:217`; python: `scripts/version_and_commit.py:131`                                                                                | Run `git status --porcelain`; if dirty, either `git stash --include-untracked` or fail explicitly. Verify clean after the rebase.                                                                                                              |
| **No push retry on non-fast-forward.** If remote `main` advances between fetch and push, `git push` is rejected; only Rust handles this with retry + `pull --rebase`.                                                                                    | vk-bot-desktop, **js**, **python**, **csharp** templates (Rust is correct) | js/vk: `scripts/version-and-commit.mjs:268`; python: `scripts/version_and_commit.py:219`; csharp: `scripts/version-and-commit.mjs:403-404`                              | Adopt the Rust template's retry loop (`scripts/version-and-commit.rs:498-525`): try push up to N times, on failure run `git pull --rebase origin <branch>` then retry.                                                                         |
| **No verification that `version-and-commit` left the repo in the expected state.** Caller has no way to confirm the new commit is on remote `main` before downstream jobs check it out.                                                                  | vk-bot-desktop (most acute), **js** template                               | vk-bot-desktop js workflow: `release_target` step at `.github/workflows/js.yml:472-481` resolves `origin/main` SHA without verifying it matches the just-pushed commit. | After push, re-fetch and assert `git rev-parse HEAD == git rev-parse origin/main`. Emit a `committed_sha` output and have downstream `desktop-build` check out that exact SHA.                                                                 |

The **intra-workflow race** between `release` and `preview-regen` in vk-bot-desktop's `js.yml` (both push to `main`, no `needs:` link) is repo-specific — no template has a preview-regen-style job — but it is what made the silent-push bug catastrophic in production. Fixed in this PR by adding `needs: [release, instant-release]` to `preview-regen`.
